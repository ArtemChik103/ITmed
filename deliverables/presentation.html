Постановка 02 / 08

Что именно нужно было сдать

Смысл этой версии проекта не в исследовательской истории, а в понятном финальном комплекте для проверки.

  • Публичный репозиторий с актуальным кодом и README без противоречий.
  • Файл `predictions.csv` в формате `id,class` без пропусков и дублей.
  • Архив `results_test_done.zip` с `summary.csv` и детальными `json/txt` по каждому объекту.
  • Презентация, которую можно быстро показать за 5 минут и так же быстро распечатать в PDF.
Почему архив текстовый
Сборка проще и стабильнее, чем рендер набора jpg
Проверка удобно валидировать автоматически по CSV и JSON
Защита эксперт открывает summary и сразу идет к нужному id
Риск меньше шансов сломаться на шрифтах, картинках и предпросмотре
Архитектура 03 / 08

Classifier-first runtime

Диагноз дает классификатор. Explainability подключается рядом, но не вмешивается в итоговый бинарный verdict.

01 · intake

DICOM вход

Backend принимает `.dcm`, читает pixel data, извлекает метаданные и валидирует снимок до inference.

02 · runtime

Plugin `hip_dysplasia`

Плагин считает `confidence`, `threshold`, бинарный класс и статусы загрузки runtime на уровне результата.

03 · explainability

Keypoints

Keypoint branch включается только как explainability layer в режиме `education` и не меняет диагноз.

Интерфейс 04 / 08

Два режима, два сценария

Интерфейс разделен по задаче пользователя, а не по количеству виджетов.

Режим врача

Короткая клиническая сводка

  • бинарный verdict
  • confidence и threshold
  • runtime/fallback status
  • validation warnings и DICOM metadata

Смысл режима: быстрое чтение результата без визуального шума.

Режим обучения

Explainability без подмены диагноза

  • keypoint overlay
  • расширенный JSON/PDF слой
  • пояснение статуса geometry
  • прозрачная логика ограничения

Смысл режима: показать anatomy layer и объяснить, что именно система умеет и чего не обещает.

Ограничение 05 / 08

Почему geometry отключена

В этой части мы сознательно выбрали честность вместо красивых, но невалидированных чисел.

Ключевая формулировка
Количественная geometry автоматически не рассчитывается, потому что семантика raw MTDDH keypoints пока не валидирована для клинического использования.
Что это значит practically
  • Keypoints остаются в продукте как anatomy/explainability layer.
  • UI, TXT, JSON и PDF пишут об ограничении одинаково и без заглушек.
  • Мы не расширяем clinical claim за счет недоказанной интерпретации landmarks.
Pipeline 06 / 08

Один batch по всему test_done

Никаких ручных пяти шагов. Один скрипт проходит по объектам и формирует весь экспертный пакет.

01

собрать все DICOM внутри объекта и определить `id`

02

прогнать локальный runtime и получить object-level verdict

03

сохранить `predictions.csv` и расширенный `summary.csv`

04

сохранить `reports/{id}.json` и `reports/{id}.txt`

05

упаковать `results_test_done.zip` и проверить полноту

06

экспортировать HTML-презентацию в финальный PDF

Финал 08 / 08

Что важно проговорить на защите

Главный акцент не на количестве файлов, а на воспроизводимости и честной границе применимости решения.

  • Classifier runtime выдает бинарный verdict для всех объектов test_done.
  • Explainability сохранен, но не подменяет клиническое решение.
  • Geometry не выдумывается автоматически и это зафиксировано во всех отчетах.
  • Все deliverables можно пересобрать локально из текущего репозитория.
Итог Сдача оформлена как аккуратный инженерный пакет, а не как набор временных артефактов.

HTML-файл является исходником презентации. Скрипт нужен только для экспорта этого файла в PDF, чтобы не дублировать верстку и не рисовать второй вариант вручную.
