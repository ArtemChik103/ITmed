AGENTS.md — строгие правила для агента (IT-Med)
0) Обязательный порядок чтения перед любыми действиями
Сначала прочитай (в этом порядке):

AGENTS.md (этот файл)
TZ.md
README.md
Запрещено начинать правки, пока не изучены файлы выше.

1) Где можно и нельзя писать

Запрещено изменять:
.git/**
CI/CD
Любые секреты/ключи/токены
Случайные служебные артефакты (__pycache__, временные файлы и т.д.)
Если задача требует правок вне разрешённых путей — сначала явно напиши, что это выходит за policy.

2) Стиль и качество кода (обязательно)
Никаких пустых файлов.
Никаких файлов только с import/комментариями.
Не оставлять неиспользуемые переменные и мёртвый код.
Не добавлять “заглушки”, которые ломают существующий flow.
Сохранять совместимость текущей архитектуры:
FastAPI endpoint в api/main.py
схемы в api/schemas.py
DICOM loader в core/dicom_loader.py
UI в frontend/app.py
Не делать “тихое” подавление ошибок без логирования.
Для новых API-ответов использовать pydantic-модели.
Не менять публичные контракты API без явного описания в итоге.
3) Правила для зависимостей
Не добавляй новые зависимости без необходимости.
Если добавил зависимость:
обнови requirements.txt
кратко объясни зачем
Старайся использовать уже имеющиеся библиотеки проекта.
4) Тесты и проверки (обязательно)
После изменений агент должен:

Запустить релевантные проверки (минимум):
pytest -q (или эквивалент с корректным PYTHONPATH)
Для API-изменений сделать smoke-check:
запуск FastAPI
проверка /health
Для UI-изменений проверить запуск Streamlit (без падения на старте).
Если что-то не запускается из-за окружения — явно указать причину и команду, которая упала.

5) Git-дисциплина
После каждого логически завершённого блока измененй:

git add ...
git commit -m "..."
Требования к commit message:

Кратко и по делу
Формат:
feat: ...
fix: ...
refactor: ...
docs: ...
test: ...
Не смешивать несвязанные изменения в одном коммите.

6) Что обязательно в финальном ответе агента
Краткое Summary:
что изменено
зачем
Список изменённых файлов.
Список выполненных команд проверки и их статус:
успешно
частично/ограничение окружения
ошибка
Если были ограничения среды — указать, как воспроизвести локально.
8) Минимальные требования результата (для задач “сделать рабочий минимум”)
Агент должен обеспечить: A) Рабочий API:

GET /health
POST /api/v1/analyze не падает на валидном DICOM-файле
B) Рабочий frontend:

загрузка .dcm/.dicom
вызов API
отображение результата/ошибки
C) Базовые тесты:

схемы API
plugin manager
smoke import тесты
8) Запрещено
Ломать существующие эндпоинты без миграции.
Менять формат ответа API “молча”.
Оставлять незакоммиченные изменения.
Коммитить временные/генерируемые файлы.
Пропускать раздел с проверками в итоговом отчёте.
